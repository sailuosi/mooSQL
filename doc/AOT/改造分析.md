mooSQL typeHandle 实体映射：Emit 现状与源生成器（AOT）并行方案分析
1. 总体定位
typeHandle 模块承担「DbDataReader → 实体/DTO」的物化（materialization），整体设计明显参考 Dapper SqlMapper：按列名匹配成员、支持构造函数注入、运行时按 ResultSet 列结构生成专用反序列化器、多层缓存。

核心类是 PackUp（partial，分散在多个文件），对外统一入口是：

        internal  Func<DbDataReader,DBInstance, object> GetPacker(Type type, DbDataReader reader, int startBound, int length, bool returnNullIfFirstMissing,DBInstance db)
        {
            // dynamic is passed in as Object ... by c# design
            if (type == typeof(object) || type == typeof(SooRow))
            {
                return GetSooRowPacker(reader, startBound, length, returnNullIfFirstMissing);
            }
            // ... 标量/枚举/数组走 GetStructPacker
            // ... 注册了 TypeHandler 的类型走 GetParserPacker
            return GetTypePacker(this,type, reader, startBound, length, returnNullIfFirstMissing,db);
        }
Emit 只覆盖「复杂类型」路径（普通 class/struct/ValueTuple），标量、SooRow、TypeHandler 类型均不走 Emit。

3. Emit 核心做了什么
GetTypePackImpl 用 DynamicMethod 生成 IL，签名固定为 (DbDataReader, DBInstance) → object：

        internal Func<DbDataReader, DBInstance, object> GetTypePackImpl(...)
        {
            var dm = new DynamicMethod("Deserialize" + Guid.NewGuid().ToString(), returnType, new[] { typeof(DbDataReader), typeof(DBInstance) }, type, true);
            var il = dm.GetILGenerator();
            if (MapperUntils.IsValueTuple(type))
                GenerateValueTupleDeserializer(...);
            else
                GenePackFromMap(...);
            return (Func<DbDataReader, DBInstance, object>)dm.CreateDelegate(funcType);
        }

5. AOT 不兼容点（需源生成器解决的）
5.1 硬阻断
System.Reflection.Emit.DynamicMethod：GetTypePackImpl 核心，Native AOT 下不可用
Emit 内的 MakeGenericMethod：如 GetFieldValue<T>、TypeHandlerCache<T>.Parse
5.2 软阻断（AOT 可运行但需改造或降级）
AddTypeHandlerImpl 运行时注册：MakeGenericType + MethodInfo.Invoke
DefaultTypeMap 运行时反射：PropertyInfo/FieldInfo/ConstructorInfo 在 AOT 中需 [DynamicallyAccessedMembers] 或编译期已知
EntityContext.getEntityInfo：运行时解析实体特性，源生成器应在编译期读取同一套 attribute 规则
匿名类型：_type.IsAnonymous() 特殊构造逻辑，源生成器难以覆盖
5.3 架构层面的核心矛盾
当前 Emit 是「按运行时 ResultSet schema 特化」，而非「仅按实体类型特化」：

            var names = Enumerable.Range(startBound, length).Select(i => reader.GetName(i)).ToArray();
            ITypeMap typeMap = GetTypeMap(type);
            // ...
            var ctor = typeMap.FindConstructor(names, types, this);  // types 来自 reader.GetFieldType
            var members = names.Select(n => typeMap.GetMember(n)).ToList();
这意味着：

同一 User 实体，SELECT * 与 SELECT 部分列，会生成 不同的 IL
列顺序、列类型（如 int vs long）影响构造函数选择与转换分支
源生成器若只按实体类型生成，无法 1:1 复刻现有行为，除非额外约定 schema 或做运行时校验/降级
6. 并行新增源生成器的推荐架构（不动现有 Emit）
6.1 设计原则
Emit 路径零改动，继续服务 net451/net462、动态 SQL、任意列组合
新增 Materializer 抽象层，在 GetTypePackImpl 之前尝试解析预生成实现
源生成器只覆盖 可静态确定的子集；匹配失败则走现有 Emit（非 AOT）或 AOT 下的 解释型/反射降级（若你允许）
6.2 建议的分层
                    GetPacker / GetTypePackImpl
                              │
              ┌───────────────┴───────────────┐
              │   IMaterializerResolver       │  ← 新增，无 Emit 依赖
              └───────────────┬───────────────┘
                              │
         ┌────────────────────┼────────────────────┐
         ▼                    ▼                    ▼
  SourceGenerated      Emit (现有)          Fallback
  Materializer         PackUp.emit.cs       (AOT 专用慢路径，可选)
Resolver 输入：(Type, columnHash 或 columnNames[], columnTypes[])
Resolver 输出：Func<DbDataReader, DBInstance, object> 或强类型 Func<DbDataReader, DBInstance, T>

6.3 源生成器触发范围（建议分阶段）
Phase 1 — 高价值、低歧义

带实体特性、通过 IEntityAnalyseFactory 可解析的 POCO 实体
无参构造 + 属性/字段赋值（与 Repository/SQLClip 典型用法一致）
列绑定：编译期按 DbColumnName / 属性名 / Alias 建立 列名 → 成员 字典
运行时按 列名 查字典（仍读 reader，但不 Emit），列顺序可变
Phase 2 — 扩展

有参构造（需编译期固定「期望列名+顺序」或生成多版本）
ValueTuple 物化（对应 GenerateValueTupleDeserializer）
与 Settings.ApplyNullValues、returnNullIfFirstMissing 行为对齐
Phase 3 — 可选/困难

运行时 SetTypeMap 自定义 ITypeMap
运行时 AddTypeHandler 新类型
匿名类型、动态 object、Multi-map splitOn 组合
6.4 与现有缓存的衔接
TypePackerCache 的 DeserializerKey 已包含列名+类型 hash，源生成路径可以：

方案 A：Resolver 返回的 delegate 仍放入 TypePackerCache（最小侵入）
方案 B：SG 生成「按实体类型唯一」的 materializer，内部自行按列名匹配，弱化对 column hash 的依赖（AOT 场景更友好，但与现有 Query cache 语义略有差异）
建议 AOT 默认 方案 B + 列名匹配，非 AOT 保持现有 hash 缓存；两者通过 Resolver 统一。

11. 结论
Emit 集中在 PackUp.emit.cs，是 Dapper 式「运行时按 ResultSet 特化」的实现；周围 DefaultTypeMap、TypePackerCache、ReadTypeConverter 是可复用的映射与转换基础设施。
并行加源生成器完全可行，最佳切入点是新增 Resolver 层，在 GetTypePackImpl 前短路，不修改现有 Emit 文件逻辑。
最大难点不是 IL → C# 翻译，而是 运行时 schema 特化 vs 编译期实体特化 的语义差异；建议 SG 采用「实体 + 列名」模型，Emit 继续兜底动态场景。
已有 EntityContext / EntityColumn 体系 是 SG 的输入金矿，Generator 应复用同一套 attribute 解析规则（编译期实现 BaseEntityAnalyseFactory 的等价逻辑）。
多目标共存：net451/462 仅 Emit；net6+ 默认 Emit，启用 AOT 时优先 SG；长期可让 SG 成为 net8+/AOT 的主路径，Emit 作为非 AOT 动态 ORM 的兼容层。

## 12. 已确认落地决策（2026-05）

| 项 | 决策 |
|----|------|
| 兼容性 | 向前兼容；**不修改**现有 Emit 逻辑（`PackUp.emit.cs`） |
| **AOT 开关** | `MooClient.EnableAot`，**默认 false**；关闭时全 TFM 走 **Emit**，**不调用 SG** |
| 演进路线 | 短期 SG 与反射并行；稳定后以 **SG 为主**，反射为兜底 |
| 调度链（AOT 关，默认） | **Emit**（与现有行为完全一致） |
| 调度链（AOT 开，net6+） | **SG → 反射**（Emit 不参与） |
| 调度链（AOT 开，net451/462） | 仅 **反射**（无 Analyzer） |
| Schema 绑定 | **实体中心 + 列名匹配** |
| SG 未命中 | 降级为 **反射物化器** |

### 12.1 调度流程

```
GetTypePacker → TypePackerCache → MooClient.EnableAot?
  ├─ false（默认）→ GetTypePackImpl (Emit)
  └─ true → MaterializerResolver
              ├─ MaterializerRegistry 命中 → SourceGenerated
              └─ 未命中 → ReflectionMaterializer
```

### 12.2 MooClient AOT 开关

- 属性：`MooClient.EnableAot`（默认 `false`）
- Builder：`BaseClientBuilder.useAotMode(bool enable = true)`
- 切换时：`MapperCache.PurgeQueryCache()`（连带清除 `TypePackerCache`）
- `EnableAot == false` 时即使引用 Generator 也**不**查询 `MaterializerRegistry`

### 12.3 分阶段实施

| 阶段 | 内容 |
|------|------|
| P0 | 文档、Resolver、`EnableAot` 开关 |
| P1 | `ColumnValueReader`、`ReflectionMaterializer` |
| P2 | `mooSQL.Pure.Generators` MVP |
| P3 | 行为对齐测试、Native AOT 冒烟 |
| P4（后期） | 扩大 SG 覆盖；SG 为主 |

### 12.5 包依赖关系

- **mooSQL.Pure**：仅运行时（`GenerateMaterializerAttribute`、`MaterializerRegistry`、反射物化器等），**不引用** Generator
- **mooSQL.Pure.Generators**：独立 Analyzer 项目/NuGet，由**定义实体的业务项目**自行引用
- 源生成器扫描的是**当前编译项目**中带 `[GenerateMaterializer]` 的类型，Pure 编译时无实体可生成，故 Pure 引用 Generator 无意义

### 12.6 物化器注册（双层）

| 层级 | 类型 | 写入方式 | 解析优先级 |
|------|------|----------|------------|
| Client 实例 | `MooClient.RegisterMaterializer` | 业务/Builder 显式注册 | **1（最高）** |
| 静态引导表 | `MaterializerRegistry` | SG ModuleInitializer | 2 |
| 反射兜底 | `ReflectionMaterializer` | 运行时生成 | 3 |

推荐 AOT 启动流程：

```csharp
new BaseClientBuilder(client)
    .useAotMode(true)
    .registerGeneratedMaterializers(); // 可选：静态表 → Client 实例
```

`useGeneratedMaterializerHook(Action<MooClient>)` 可在复制静态表前注入额外注册逻辑。

### 12.4 与 Emit 的差异（AOT 开时）

- 固定 **无参构造 + 属性/字段赋值**（不做按 reader 列类型选有参构造）
- 列顺序无关；多列忽略、少列留 default
- 运行时 `SetTypeMap` / 动态 TypeHandler 在 AOT 模式下行为受限
