### 1.2 已实现（与设计一致或接近）

#### 架构与职责划分

- **主从组挂在 `MooClient`**：`ConcurrentDictionary<int, MasterSlaveGroup>`，`configureGroup` / `getGroup`（`MooClient.cluster.cs`）。
- **`DBInsCash` 为实例池**：`getInstance` + `OnInstanceRetrieved` 懒探活 + `ApplyProactiveFailoverIfNeeded` 超前选举（`DBInsCash.health.cs`）。

### 1.4 不一致点（已实现但行为/结构与文档不符）

| # | 设计预期 | 实际实现 | 影响 |
|---|---------|---------|------|
| 1 | 读从库需 `.useReadReplica()` 或明确的组级读写分离开关 | `ResolveTargetInstance`：`PreferReadReplica == true` **或** `GetGroup != null` 即走 `ResolveRead` | **只要注册了主从组，所有 `Select` 默认读从**（除非 `MasterOnly`）；与 §7.10 文案不完全一致，但与 §2.5「组级默认」可并存——**文档内部矛盾，实现偏向「组注册即读写分离」** |
| 2 | `RouteResolver` 读 fallback 用 `Options.ReadFallbackToMaster` | `fallback = ov?.ReadPolicy != null ? group.ReadFallbackToMaster : Options.ReadFallbackToMaster` | 仅当 GroupOverride **设置了 ReadPolicy** 才用组级 `ReadFallbackToMaster`，否则用全局；**与「组级 ReadFallbackToMaster 字段」直觉不符** |
| 3 | 选举候选 `CanFailover`（含 Health） | `FailoverPolicy` 用 `HotStandby && WriteEnabled` + `IsInstanceHealthy`，**未调用 `SlaveMember.CanFailover`** | 逻辑等价性大体成立，但 **DRY 与文档引用不一致** |
| 4 | 执行期读路由不修改 Builder 锚点 | `BeginExecutionRouting` **临时改 `DBLive`**，finally `EndExecutionRouting` 恢复 | 对单次 `ExecuteCmd` 正确；**`useFailover` 会永久改 Builder.DBLive**（设计允许） |
| 5 | `ImmediateOnFailure` 解析 FailoverMode | `TryImmediateFailoverAndRetry` 用 `RouteContext.FailoverOverride ?? group.FailoverMode`，**未走 `ResolveFailoverMode`（含 GroupOverride）** | 组级 `GroupOverride.Failover` 对 Immediate 重试**可能不生效** |
| 6 | `HealthProbeScheduler` 周期 | 设计关联 `RecoveryInterval`；实现 **固定 30s**，且与 `DBHealthOptions.RecoveryInterval` 脱节 | 恢复探测节奏不可 per-DB 配置 |
| 7 | `ReadRouteSelector` 轮询 | 全局静态 `_roundRobin`，**非 per-group** | 多组共享轮询计数，负载均衡略失真 |
| 8 | `WeightedRandom` | 每次 `new Random()` | 高并发下分布与性能非最优（业内常用线程安全 RNG 或轮盘指针） |
| 9 | `promoteMaster` | 可变 `group.Master` 引用 | 设计称 Master 为配置锚点；**运行时可变**，与「仅 RouteContext 临时指定」并存，运维需知副作用 |
| 10 | `electIfUnavailable` 事件触发 | 仅当 `elected` 健康且非 current 时 `FireFailover` | 选举失败返回病态 current 时**无事件**，符合「无切换」但可观测性弱 |

## 2. 逻辑冲突与设计合理性

### 2.1 文档内部张力（非纯代码 bug）

1. **组级默认读路由 vs Builder 临时读从**  
   - §2.5 优先级 4：组级 `ReadPolicy` 参与解析。  
   - §7.10：`.useReadReplica()` 仅当次 query 走从库。  
   - 实现采用「**有组则 Select 走路由**」，等价于应用级读写分离一旦 `configureGroup` 即生效；`useReadReplica()` 更像显式强化/文档化意图。  
   - **建议**：在设计文档中明确二选一或分层：**(A)** 组 `ReadPolicy != MasterOnly` → 全局读分离；**(B)** 仅 `useReadReplica()` → 临时读从。

2. **`getInstance` 超前选举 vs 读路由分离**  
   - 设计正确：`getInstance` 只保障**可写**实例，读从仍在 `ExecuteCmd`。  
   - 实现一致：仅 `Unavailable` 时选举，不误把读从库当作写库返回。  
   - **合理**。

3. **`useFailover` 永久绑定 `DBLive` vs `EndExecutionRouting` 临时切换**  
   - 灾切后 Builder 锚点变更，后续 SQL 在同 Builder 上延续新实例——符合「单笔/单批激进灾切」场景。  
   - 若业务复用 Builder 做普通查询，可能**意外继续走热备方言/连接**；需在 API 文档强调 `resetRoute` / 新建 Builder。

### 2.2 潜在运行时冲突

| 场景 | 风险 | 严重度 |
|------|------|--------|
| 事务中 `DBLive` 与 Session 物理连接不一致 | 事务锁定 Session 实例，设计禁止 Failover；实现已禁止 | 低（已防护） |
| `ImmediateOnFailure` 重试 `ExecuteCmd` | 重入 `BeginExecutionRouting`，若 RouteContext 仍指向死实例可能二次失败 | 中（选举成功则缓解） |
| 双写 + `ModifyMediator` | `SkipAsyncReplication` 在 `EnableDualWrite` 时设置 | 低（已显式跳过） |
| 双写多方言 | 同一 `SQLCmd` 多库执行，方言不同时 SQL 错误 | 高（设计已警告，未实现重编译） |
| `getInstance(0)` 返回热备 B，但 `ResolveRead` 选从库 C | 锚点 position 仍为 0，读路由按组配置，**与写锚点实例不同** | 低（符合读写分离） |
| 异步 API 无 Failover/健康闭环 | 生产若主走 `ExecuteCmdAsync`，HA 能力弱于同步 | 中 |

### 2.3 设计合理性评价

**合理之处**

- **异步复制与路由正交**：`ModifyMediator` 管「写后同步」，cluster 管「连谁」——边界清晰，避免一个类承担两种一致性模型。
- **超前灾切在 `getInstance` / `useFailover`**：减少「取库后首条 SQL 才失败」窗口，符合连接池/Session 最佳实践。
- **无组级 `ActiveMaster`**：避免脑裂后锁死错误主库；连续选举模型简洁。
- **Executor 级 `RouteContext`**：与 `session` 事务模型对称，Repository/Clip 只传 Executor，学习成本低。

**可商榷之处**

- **`RouteResolver` 与 `FailoverPolicy` 均为可独立调用的静态/实例逻辑**，但选举入口又在 `MooClient`——职责链略长，新人需跟三条线。
- **健康状态与路由状态未统一**：`Unavailable` 靠探活/执行失败设置，**连接池内陈旧连接**可能直到执行才暴露。
- **`promoteMaster` 改配置主**：运维有用，但与「配置锚点不变」的纯函数式路由混用，长期易混乱；更稳妥是仅文档化 + 外部配置热更新。
- **全局 `RoundRobin` 计数器**：多租户/多组场景下公平性不足，宜 per-group 或 per-slave 环。

## 3. 设计模式与代码结构

### 3.1 采用的模式（映射）

| 模式 | 体现 | 评价 |
|------|------|------|
| **策略模式** | `ReadRoutePolicy`、`FailoverMode`、`DualWriteErrorPolicy` | 清晰，易扩展 Custom |
| **责任链/管道** | `ExecuteCmd`：健康 → 路由 → 执行 → 恢复 DBLive | 边界明确 |
| **外观** | `MooClient.configureGroup` / `electIfUnavailable` | 业务入口简洁 |
| **上下文对象** | `SQLRouteContext`、`FailoverContext` | 符合「当次作用域」语义 |
| **部分类 partial** | `DBInsCash`、`DBExecutor`、`SQLBuilder`、`MooClient` 分文件 | 利于按特性拆分，但跨文件跳转成本高 |
| **注册表** | `_masterSlaveGroups` | 标准 |

### 3.2 单一职责（SRP）

| 类型 | 职责 | SRP |
|------|------|-----|
| `DBHealth` | 单实例状态机 | ✅ |
| `FailoverPolicy` | 选举算法 | ✅ |
| `RouteResolver` | 读/写/双写解析 | ✅ 略宽（含候选过滤） |
| `DBExecutor` | 执行 + 事务 + 健康门禁 + 路由 + Immediate 重试 | ⚠️ **偏重**；`DBExecutor.route.cs` / `health.cs` 已拆分 partial，仍是一类多因 |
| `SQLBuilder` | SQL 构建 + 路由链式 API + 双写分支 | ⚠️ `SQLBuilderDymatic` 内嵌 fan-out 判断 |
| `MooClient` | 全局客户端 + 主从注册 + 选举 + 事件 | ⚠️ cluster 部分尚可，整体 Client 仍偏大 |

### 3.3 内聚与耦合

**高内聚**

- `pure/src/ado/data/cluster/` 包内聚主从域模型。
- `health/` 独立包，通过 `DBInstance` 挂载耦合可控。

**耦合点**

- `RouteResolver` → `DBInsCash.getInstance`：解析中拉实例，**可能递归触发探活/超前选举**（读路径 `getInstance(TargetPosition)`）。
- `DBHealth.DoDefaultPing` → 新建 `DBExecutor` 执行 SQL：健康层依赖执行层，**存在循环依赖风险**（靠 `SkipHealthCheck` 打断）。
- `SQLBuilder` 直接调用 `MooClient.electIfUnavailable`：Builder 知晓 Client 集群细节，**抽象泄漏**。

### 3.4 规范与可维护性

| 项 | 观察 |
|----|------|
| 命名 | 与项目一致（小写方法 `getInstance`、`configureGroup`） |
| 可见性 | `RouteResolver`、`FailoverPolicy` public，设计建议 internal，**封装不足** |
| 异常 | 路由失败 `NoReadableReplicaException`；其余多包装为 `Exception`，与设计「可观测」部分对齐 |
| 测试 | 核心选举/读路由有单测；缺集成测（真实 TCP、事务+Failover、DualWrite 部分成功） |
| 空 catch | `loadDBCongfig`、`Probe` best-effort 多处吞异常，**不利于生产排障** |

## 4. 高层次对比：C# 生态与类似库

### 4.1 业内常见分层（抽象对比）

```
┌─────────────────────────────────────────────────────────────┐
│ 应用 / ORM (EF Core, FreeSql, SqlSugar, 自研 mooSQL)         │
├─────────────────────────────────────────────────────────────┤
│ 路由层：读/写分离、Hint、Interceptor、RouteContext           │
├─────────────────────────────────────────────────────────────┤
│ 连接层：连接串、Always On、驱动负载均衡                       │
├─────────────────────────────────────────────────────────────┤
│ 数据复制：异步复制、双写、GTID（DB 或中间件）                  │
└─────────────────────────────────────────────────────────────┘
```

mooSQL **同时覆盖路由层 + 应用侧异步复制插件**，在自研 ADO 封装库中属于**较完整**的一类；与 EF/Dapper 默认能力相比，更接近 **SqlSugar / FreeSql** 的主从方案，而非 EF Core 原生能力。

### 4.2 与主流 C# 方案对比

| 方案 | 读/写路由 | Failover | 健康检查 | 双写/复制 | 与 mooSQL 差异 |
|------|----------|----------|----------|-----------|----------------|
| **Entity Framework Core** | 无内置 RW；需第三方或手动 `DbContext` | 连接弹性（重试），非主从选举 | 无 | 无 | mooSQL **执行期路由 + 超前选举**更深；EF 依赖数据库 Always On / 外部代理 |
| **Dapper** | 无 | 无 | 无 | 无 | mooSQL 在 ADO 之上自建 HA 栈 |
| **SqlSugar** | `SlaveConnectionConfigs`，查询自动走从库 | 相对简单，偏配置切换 | 较弱 | 主从同步多为 DB 层 | mooSQL **显式 `RouteContext` + Failover 模式枚举**，控制粒度更细；SqlSugar **上手更快** |
| **FreeSql** | `[Slave]` / `UseSlave()` 读写分离 | 支持读写分离配置 | 一般 | 无应用双写 fan-out | mooSQL **DualWrite + ModifyMediator** 区分同步双写与异步复制，模型更细 |
| **MySqlConnector 连接串** | `LoadBalance` 等 | 驱动级 | 无应用语义 | 无 | mooSQL 在**应用语义**（热备选举、读策略）上更强 |
| **ShardingSphere-Proxy / JDBC 路由**（Java 参考） | 成熟读写分离 +  Hint | 强 | 强 | 中间件层 | mooSQL 轻量、嵌入式，**无独立代理进程** |
| **MassTransit / 事件驱动双写** | N/A | N/A | N/A | 最终一致 | mooSQL `DualWrite` 为**同步 SQL fan-out**，延迟低但一致性靠业务 |

### 4.3 mooSQL 的差异化与定位

**优势（相对典型 C# 轻量 ORM/ADO 封装）**

1. **写后异步复制（`ModifyMediator`）与读写路由解耦**——类似 Canal/Debezium 思路的「轻量内嵌版」，适合已有 mooSQL 栈的项目。
2. **`SQLRouteContext` 与事务 `session` 对称**——比「全局静态读写分离开关」更适合多租户、报表临时走从库。
3. **超前灾切 + 连续选举 + 无 `ActiveMaster`**——比仅「连接串 failover partner」更贴近应用可写实例语义。
4. **bool 组合能力（ReadReplica + HotStandby + DualWrite）**——比单一 `Role=Slave` 枚举更贴近运维现实。

**短板（相对 SqlSugar/FreeSql/中间件方案）**

1. **生态与文档**：商业/社区案例、可视化运维弱于 SqlSugar。
2. **异步路径 HA 不完整**：现代 ASP.NET 大量 async，需补齐 §1.3。
3. **无独立 Sidecar**：路由逻辑与业务进程耦合，升级需发版（中间件方案可独立升级）。
4. **双写/多方言**：FreeSql/SqlSugar 通常假设同构从库；mooSQL 设计已提示，实现未跟上。

## 5. 结论与建议优先级

### 5.1 结论

主从与多库的**核心架构（MooClient 注册组、RouteContext、超前灾切、无 ActiveMaster、健康+选举+读路由）已实现**，与设计文档 2026 审计目标**大体对齐**。差距集中在 **P5 细化项（超时、恢复间隔）**、**异步路径 parity**、**DualWrite 高级场景** 以及 **文档与实现关于「何时走读从」的语义统一**。

### 5.2 建议修复/澄清优先级

| 优先级 | 项 |
|--------|-----|
| P0 | 统一设计文档：组注册后 Select 是否默认读从；同步更新 §7.10 |
| P0 | `ExecuteCmdAsync` 对齐健康标记 + `ImmediateOnFailure` + `MarkHealthSuccess` |
| P1 | `DoDefaultPing` 应用 `PingTimeoutMs`；Scheduler 使用 `RecoveryInterval` |
| P1 | `TryImmediateFailoverAndRetry` 使用 `ResolveFailoverMode` |
| P1 | 修复 `ReadFallbackToMaster` 条件（`ov?.ReadPolicy` → 应优先 `group.ReadFallbackToMaster`） |
| P2 | `exeNonQueryAsync` 双写；`RouteResolver` 改 internal |
| P2 | per-group RoundRobin；`FailoverPolicy` 使用 `CanFailover` |
| P3 | DualWrite 多方言重编译；复制延迟选举；`IOptions` 化配置 |
