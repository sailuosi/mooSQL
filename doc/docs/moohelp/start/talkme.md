回到ORM这块，.Net领域的ORM主体功能方向，大概有 【特性】驱动的实体查询，Linq表达式驱动的动态查询，对ADO.NET进行扩展的基础路线，以及相对冷门的个性化查询设计类别，如无实体查询、动态对象、类MyBatis的xml映射配置、及古早的基于父类约束加内置的表结构字段的方式。而现代ORM主要以，前2个为主，代表的有官方选手EFCore、国产巨星Sqlsugar，及国外的Linq2db，FreeSQL等。剩下的，主要是冷门的个性化路线，或是功能单薄因为某些原因没有发展起来，或是其他原因，多数维护的都较为冷清了，有的已数年不更新。

mooSQL当前核心数据库层抽象、执行层、及SQL编织器层都已成熟，在多个项目中使用。基于实体类的动态查询也主体也已完成，后续会继续完善。在通用的Queryable为代表的动态Linq方面，其核心支撑已实现。Linq版的语法在实现后，却发现存在着与构想较大的落差，比如语法与SQL差异大、Join构建不便、复杂子查询难以实现、部分基于c#方法的转换为SQL逻辑时存在落差、模糊地带，尤其是条件复杂时。让笔者放弃实现更多linq版功能的最后一根稻草，是约束。不明白LINQ原理的小白，非常容易把Linq实际不支持转为SQL的C#方法，都塞到委托里，这简直是个黑洞！

在当前落地的业务系统中实际应用时，SQLBuilder发挥了远超出设想的活力，这块想细说一下，大家可以留言交流。

初步使用你或许觉得，SQLBuilder不就是一个拼接SQL字符串的类，有什么特殊？我在函数里用StringBuilder一样能做到。然而，当它带着mooSQL的一整套基础能力（数据库方言抽象、驱动集成、切面、事件、监听器、缓存、自定义注解、仓储、实体快捷查询和修改、实体动态查询和修改）集成起来后，就是一个瑞士军刀。

举个例子，MyBatis最为常用的在XML的配置功能有：

1、实体映射，这块相当于mooSQL的实体特性功能，可为SQLBuilder自动产出转换使用，已集成；

2、SQL语句碎片复用，这个在mybatis超常用，一般复杂的SQL，会用id标记起来，然后在其他地方引用，这点非常好。可以说是弱点就是因为在XML配置，难以利用编程语言强大的过程控制、函数等。然而SQLBuilder作为集成了构造上下文的载体，可以任意在函数、类中进行逻辑的插入、修改，利用c#强大的扩展函数，轻松集成项目的特殊逻辑。这点是笔者项目中用的最爽的地方，最典型的在SQLBuilder上构建起的权限过滤逻辑实现，比如。

var kit=DB.useSQL(0);//取得一个SQLBuilder实例
kit.select(xxx)....from... join. ...where...// 吧啦吧啦一对业务表的查询逻辑

            .xxxx //其他业务+权限的个性化逻辑，比如业务这里写时，提供业务上的参数
})

3、事务管理。在java领域，事务常用方法级注解实现，然而却容易埋坑，当一个事务方法，被另一个事务调用时，其行为难以预测。.Net领域内ORM，事务这块通常进行高级封装，一般只常用隐性事务，即（对一堆实体SaveChange）时，当需要显性事务时，便捷的实体保存逻辑和个性化的SQL事务，难以直接共享；当完全手工控制时，又太过复杂。

在mooSQL里，SQLBuilder本身同样承载了事务功能，当一个调用起点开启事务后，所有后续SQLBuilder本身提供的操作，以及由SQLBuilder延伸出的仓储、实体动态修改、批量插入等，通通自动串在一个事务里，轻松把事务管理起来。这点也是在项目实践中用的比较爽的一点。

4、日志追查、错误输出、慢SQL监控等方面，作为一个本身就聚焦SQL的工具，这方面自然是重点。项目开发期和后续运维期，SQL执行情况，都是关注中的重点，提前管理日志，完善的日志，常常对故障早期发现、故障判断、原因追查，都有很大的裨益。

5、特色的查询实体功能，允许定义特殊的查询实体，表来源可以是多表join，字段可以是多个表的综合，可以是SQL函数。这里相当于是把mybatis的join查询实体结果映射给支持了。


在不足和未来方面，便捷性生态、实体动态查询增强、分库分表导航查询等高级功能会逐步推进。数据库的迁移、自动初始化表结构和种子数据等需要加入。特殊场景扩展方向上，有读写分离、主从库同时写入、业务实体版本支持功能等。数据库支持上，还需要增加更多的数据库内置支持（虽然自己定义也很容易）

在开发方面，其实一个足够自由、能够操作底层的ORM，对于一个流程引擎、开发平台是十分重要的。必须得能够切入到细粒度的控制，才能更好的为性能、设计服务。
